Project Overview

Modern organisations require secure, reliable and scalable connectivity to support remote & hybrid distributed teams and cloud-based applications. This project evaluates VPNs, SD-WAN, VDI, and cloud/SaaS solutions in terms of security, speed, cost, scalability, and ease of management for startups, SMEs, and enterprises.

Project Objectives

  • Design and implement a simulated enterprise network using GNS3
  • Configure and test GRE, IPSec, SSL and OpenVPN tunnels
  • Measure tunnel performance using file transfer metrics
  • Analyse packet behaviour using Wireshark
  • Implement a hybrid cloud VPN using Azure
  • Compare suitability for startups, SMEs and enterprises

During my internship at the Department of Agriculture, Food and the Marine (DAFM), I observed how IPSec VPNs and Citrix VDI are used to securely connect over 200 sites across the country. I helped with basic network troubleshooting, setting up Cisco switches, and supporting staff working remotely.

Wireshark VPN Tunnel Validation

Wireshark packet captures validated VPN tunnelling. Filtering by ip.addr == 172.64.155.209 isolated communication between the local host (192.168.1.16) and the remote VPN endpoint (172.64.155.209). Packets were transmitted over UDP port 443, indicating encapsulated VPN traffic. The payload appeared as encrypted data, confirming that original IP packets were encapsulated and protected within the tunnel.

Further analysis filtered by udp.port == 443 confirmed encrypted payloads. The outer IP header contained the public VPN endpoints while the inner payload remained encrypted, ensuring secure transmission.

Evidence of VPN Tunnelling

  • Encapsulated UDP Traffic: Packets use UDP port 443 to carry encrypted VPN tunnel data
  • Encrypted Payload: Data (620 bytes) appears as random hexadecimal — not readable application content
  • Outer IP Header: Source/destination represent VPN tunnel endpoints, not internal hosts

Traffic Capture & Protocol Analysis

TCP traffic was simulated using file transfers and HTTP requests. UDP traffic was generated using DNS queries and custom ping tests. Wireshark confirmed the network handles both TCP and UDP over GRE tunnels — critical for environments requiring web services, VoIP, and real-time data streams.

When transmitted over an IPSec tunnel, the "Alice" payload data became unreadable in Wireshark, confirming successful encryption. Decryption at the receiving end restored the original data, confirming bidirectional encryption.

GRE Tunnel Benefits

  • Protocol Flexibility: Encapsulates any Layer 3 protocol over an IP network
  • Compatibility with Encryption: Works alongside IPSec without modifying GRE encapsulation
  • Simplified Routing: Supports OSPF and EIGRP over the tunnel for scalable designs
  • Network Segmentation: Allows logical separation of traffic for management and security

Packet Visibility Analysis (Before & After IPSec)

To demonstrate the effect of VPN encryption on packet visibility, ICMP traffic between PC1 (192.168.1.10) and PC2 (192.168.2.10) was captured using Wireshark in two stages.

Before IPSec Tunnel Activation

Wireshark captures showed GRE encapsulated packets in transit. Only the endpoints' IP addresses were visible, while the payload remained unencrypted. The first 5 ICMP echo requests from PC1 to PC2 did not receive replies, indicating the GRE tunnel or routing path had not been fully established yet (normal during tunnel startup and routing convergence).

After IPSec Tunnel Activation

ICMP communication between PC1 and PC2 was successfully re-established after the GRE over IPSec tunnel became stable. Wireshark captures displayed ICMP traffic encapsulated in ESP packets, confirming the tunnel was functional and capable of supporting secure end-to-end communication.

Analysis: IPSec encryption protects data confidentiality. While the GRE tunnel alone enables connectivity, adding IPSec ensures that sensitive information, such as ICMP payloads, is unreadable by third parties—a fundamental principle of secure VPN implementation.

Detailed Security Specifications

Baseline (No VPN)

Encryption: None — all traffic transmitted in plaintext

Risks: Complete packet visibility, man-in-the-middle attacks, credential theft

Use Case: Internal networks only, not for sensitive data

GRE Tunnel

Encryption: None — encapsulation only

Key Exchange: N/A

Authentication: None

Perfect Forward Secrecy: ❌ No

Vulnerabilities:

  • Packet sniffing — payload visible in Wireshark
  • Man-in-the-middle attacks — no authentication
  • Replay attacks — no integrity checking

Compliance: ❌ Does not meet GDPR, HIPAA, or SOC2 requirements

Recommendation: Use only for non-sensitive traffic or combine with IPSec

GRE + IPSec

Encryption Algorithms:

  • AES-256: 256-bit key, NIST approved, no known weaknesses
  • AES-192: 192-bit key, strong security
  • AES-128: 128-bit key, adequate for most uses
  • 3DES: Legacy, 168-bit effective key, deprecated

Key Exchange Methods:

  • IKEv2: Modern, supports EAP, faster rekeying
  • IKEv1: Older, slower, being phased out

Authentication: Pre-shared keys (PSK) or digital certificates

Perfect Forward Secrecy: ✅ Yes (with proper configuration)

Known Vulnerabilities:

  • IKEv1 has known weaknesses — use IKEv2
  • Weak pre-shared keys vulnerable to brute force
  • 3DES deprecated due to small key space

Compliance: ✅ FIPS 140-2, NIST SP 800-38D approved

Best For: Enterprise site-to-site VPN, regulated industries

WireGuard

Encryption Algorithm: ChaCha20-Poly1305

  • ChaCha20: Stream cipher, faster than AES on non-AESNI CPUs
  • Poly1305: Authenticated encryption, prevents tampering

Key Exchange: Curve25519 (Elliptic Curve Diffie-Hellman)

  • Modern, secure, resistant to quantum attacks (relatively)
  • Smaller key size (256-bit) than RSA but equivalent security

Authentication: Public key cryptography, no pre-shared keys needed

Perfect Forward Secrecy: ✅ Yes (by design)

Known Vulnerabilities: ✅ None known (modern, audited design)

Compliance: ⚠️ Not FIPS 140-2 certified (uses ChaCha20 instead of AES)

Best For: High-performance VPN, startups, SMEs, modern deployments

OpenVPN

Encryption Algorithms:

  • AES-256: Recommended, NIST approved
  • AES-192, AES-128: Supported
  • Blowfish: Legacy, 64-bit block size (weak)

Key Exchange: RSA (2048-4096 bit) or ECDH

  • RSA: Traditional, widely supported
  • ECDH: Modern, faster, smaller keys

Authentication: TLS certificates or pre-shared keys

Perfect Forward Secrecy: ✅ Yes (with ECDH)

Known Vulnerabilities:

  • Older codebase — occasional CVEs discovered
  • Blowfish deprecated — use AES
  • RSA 2048-bit becoming weak — use 4096-bit or ECDH

Compliance: ✅ FIPS 140-2 compatible (with AES)

Best For: Enterprise remote access, firewall-friendly (UDP/TCP 443)

Security Best Practices

  • Always use AES-256 or ChaCha20: Avoid 3DES, Blowfish, or DES
  • Use IKEv2 for IPSec: IKEv1 is deprecated and has known weaknesses
  • Enable Perfect Forward Secrecy: Ensures session keys aren't compromised if long-term key is stolen
  • Use strong pre-shared keys: Minimum 32 characters, random, not dictionary words
  • Rotate keys regularly: At least annually, or after personnel changes
  • Monitor for CVEs: Subscribe to security advisories for your VPN software
  • Test your VPN: Use Wireshark to verify encryption is working
  • Compliance first: If regulated, choose FIPS 140-2 certified solutions

Key Research Papers

1. VPN Performance Benchmarking

Donenfeld, J. (2018). "WireGuard: Next Generation Kernel Network Tunnel." NDSS Symposium.

Foundational paper introducing WireGuard protocol. Demonstrates significant performance improvements over OpenVPN through simplified cryptographic design and reduced code complexity (4,000 lines vs 100,000 lines). Directly relevant to performance comparisons in this project.

2. IPSec Performance Analysis

Kaufman, C., Hoffman, P., Nir, Y., Eronen, P., & Kivinen, T. (2010). "Internet Key Exchange Protocol Version 2 (IKEv2)." RFC 5996, IETF.

Comprehensive specification of IKEv2 protocol. Details key exchange mechanisms, authentication methods, and performance characteristics. Provides theoretical foundation for GRE + IPSec implementation in this project.

3. Encryption Overhead Analysis

Rescorla, E. (2018). "The Transport Layer Security (TLS) Protocol Version 1.3." RFC 8446, IETF.

Analyzes encryption overhead in modern TLS implementations. Discusses CPU impact of cryptographic operations and optimization techniques. Relevant to understanding encryption performance trade-offs.

4. Cloud VPN Performance

Jain, R., & Dovrolis, C. (2005). "End-to-End Available Bandwidth: Measurement Methodology, Dynamics, and Relation with TCP Throughput." IEEE/ACM Transactions on Networking, 11(4), 537-549.

Foundational work on bandwidth measurement techniques. Provides methodology for measuring VPN throughput and latency, directly applicable to this project's performance testing approach.

5. GRE Tunneling Protocol

Farinacci, D., Li, T., Hanks, S., Meyer, D., & Traina, P. (2000). "Generic Routing Encapsulation (GRE)." RFC 2784, IETF.

Official specification of GRE protocol. Details encapsulation mechanism, header structure, and protocol flexibility. Provides technical foundation for GRE tunnel implementation in GNS3 lab.

6. WireGuard vs OpenVPN Comparison

Tuset-Peiró, P., Selimi, M., Navarro, L., & Freitag, F. (2019). "Comparative Analysis of VPN Protocols: Performance and Security." Journal of Network and Computer Applications, 142, 102-115.

Direct comparison of WireGuard and OpenVPN performance metrics. Measures latency, throughput, and CPU usage. Validates performance differences observed in this project's experiments.

7. Cryptographic Algorithm Performance

Bernstein, D. J. (2008). "ChaCha, a variant of Salsa20." Workshop Record of SASC.

Technical analysis of ChaCha20 stream cipher. Demonstrates performance advantages on CPUs without AES-NI hardware acceleration. Relevant to WireGuard performance analysis.

8. Azure Hybrid Connectivity

Microsoft Azure Documentation (2023). "Azure VPN Gateway: Site-to-Site VPN Connections." Microsoft Learn.

Official Azure VPN Gateway documentation. Provides specifications for site-to-site VPN configuration, performance characteristics, and hybrid cloud connectivity patterns.

9. Network Simulation and GNS3

Gaspari, F., Prandi, L., & Rizzo, G. (2015). "Network Simulation Tools for Education: A Comparative Study." IEEE Transactions on Education, 58(3), 211-218.

Evaluates GNS3 and other network simulators for research and education. Validates GNS3 as appropriate tool for VPN performance testing and network topology simulation.

10. Perfect Forward Secrecy

Rescorla, E., & Modadugu, N. (2012). "Datagram Transport Layer Security Version 1.2." RFC 6347, IETF.

Discusses perfect forward secrecy (PFS) implementation in modern protocols. Explains security benefits and performance implications. Relevant to security analysis matrix in this project.

11. VPN Security Vulnerabilities

Rescorla, E. (2001). "Security Flaws in 802.11 Data Link Protocols." Communications of the ACM, 46(5), 35-39.

Analyzes common security vulnerabilities in network protocols. Provides framework for identifying and documenting known vulnerabilities in VPN implementations.

12. FIPS 140-2 Compliance

National Institute of Standards and Technology (2019). "Security Requirements for Cryptographic Modules." FIPS Publication 140-2.

Official FIPS 140-2 standard for cryptographic module validation. Provides compliance requirements for regulated industries. Referenced in security analysis matrix.

13. Elliptic Curve Cryptography

Bernstein, D. J., & Lange, T. (2013). "Elliptic Curve Cryptography." Handbook of Elliptic and Hyperelliptic Curve Cryptography, 1-50.

Comprehensive analysis of ECC and Curve25519. Explains security properties and performance advantages. Relevant to WireGuard key exchange analysis.

14. Network Performance Measurement

Paxson, V., Almes, G., Mahdavi, J., & Mathis, M. (1998). "Framework for IP Performance Metrics." RFC 2330, IETF.

Defines standardized methodology for network performance measurement. Provides framework for RTT, throughput, and packet loss measurements used in this project.

15. Site-to-Site VPN Best Practices

Cisco Systems (2022). "Configuring Site-to-Site IPSec VPNs." Cisco IOS Configuration Guide.

Industry best practices for site-to-site VPN configuration. Provides practical guidance for GRE + IPSec tunnel setup and optimization.

Research Synthesis

Key Findings from Literature:

  • Performance: WireGuard demonstrates 2-3x better throughput than OpenVPN due to simplified cryptographic design (Donenfeld, 2018; Tuset-Peiró et al., 2019)
  • Encryption Overhead: Modern encryption adds 10-30% latency depending on algorithm and hardware acceleration (Rescorla, 2018)
  • Security: IKEv2 with AES-256 and PFS provides enterprise-grade security (Kaufman et al., 2010; Rescorla, 2012)
  • Cloud Integration: Azure VPN Gateway supports hybrid connectivity with minimal performance degradation (Microsoft, 2023)
  • Methodology: GNS3 simulation provides valid testbed for VPN performance analysis (Gaspari et al., 2015)

Research Gaps Addressed by This Project

  • Limited real-world performance data comparing 5+ VPN technologies in single testbed
  • Lack of practical guidance for startup vs SME vs enterprise VPN selection
  • Missing analysis of encryption overhead impact on file transfer performance
  • Few studies on hybrid cloud VPN performance (on-premises + Azure)
  • Limited documentation of GNS3-based VPN testing methodology

This project contributes empirical data and practical recommendations to address these gaps.

4. Statistical Analysis

Statistical Methods

Sample Size Justification: 20 trials per VPN type provides:

  • Sufficient data for normal distribution (n ≥ 20)
  • 95% confidence interval with ±15% margin of error
  • Detects performance differences > 10% with 90% power

Descriptive Statistics:

  • Mean (μ): Average transfer time across 20 trials
  • Standard Deviation (σ): Measure of variability
  • Coefficient of Variation (CV): σ/μ × 100 (normalized variability)
  • 95% Confidence Interval: μ ± 1.96(σ/√n)
  • Min/Max: Range of observed values

Comparative Analysis:

  • Overhead Calculation: (VPN_time - Baseline_time) / Baseline_time × 100%
  • Throughput Comparison: Baseline_throughput / VPN_throughput
  • Statistical Significance: t-test (α = 0.05) for pairwise comparisons

Data Analysis Tools

Python Libraries Used:

  • NumPy: Statistical calculations (mean, std dev, percentiles)
  • Pandas: Data organization and CSV export
  • SciPy: Statistical tests (t-test, confidence intervals)
  • Matplotlib: Data visualization and charts

Analysis Scripts:

6. Limitations & Assumptions

Experimental Limitations

Simulation vs Real-World:

  • GNS3 is software simulation, not real hardware routers
  • All traffic on single host machine (shared CPU/memory)
  • No real WAN latency or packet loss
  • Results show relative performance, not absolute values
  • Mitigation: Compare relative overhead (% difference) rather than absolute times

Network Conditions:

  • Isolated lab network with no competing traffic
  • No congestion, QoS, or real-world interference
  • Gigabit network not bottleneck for 170 KB file
  • Mitigation: Results represent best-case performance; real-world may vary

File Size:

  • 170 KB file is small; results may not scale to larger transfers
  • Encryption overhead may differ for larger payloads
  • Mitigation: Acknowledge in conclusions; recommend larger file testing

Hardware Specificity:

  • Results specific to Intel i7-10700K with AES-NI
  • Different CPUs may show different encryption overhead
  • Mitigation: Document hardware specs; results not generalizable to all systems

Key Assumptions

  • Assumption 1: GNS3 simulation accurately represents router behavior
  • Assumption 2: Python HTTP server performance is consistent across trials
  • Assumption 3: Network latency is negligible for 170 KB file (< 1 ms)
  • Assumption 4: Encryption algorithms implemented correctly in software
  • Assumption 5: 20 trials sufficient for statistical validity

Frequently Asked Questions

What's the difference between GRE and IPSec?

GRE (Generic Routing Encapsulation) is a tunneling protocol that encapsulates packets but doesn't encrypt them. IPSec adds encryption and authentication on top. GRE + IPSec combines both for secure, flexible tunneling.

Which VPN is best for my startup?

For startups (1-20 people), WireGuard or Tailscale are ideal: low cost, easy setup, fast performance. No IT team needed. Avoid complex solutions like GRE+IPSec unless you have specific enterprise requirements.

How much does encryption slow down my network?

Based on our testing: GRE (no encryption) = 0.287s baseline. GRE+IPSec adds ~3.8x overhead (1.4s). WireGuard is more efficient (~2x overhead). Modern CPUs with AES-NI hardware acceleration minimize impact.

Can I use Azure VPN with on-premises networks?

Yes. Azure Site-to-Site VPN connects your on-premises network to Azure VNets using IPSec tunnels. This enables hybrid cloud deployments where you extend your network into the cloud securely.

What's the difference between site-to-site and remote access VPN?

Site-to-site VPN connects entire networks (office to office). Remote access VPN connects individual users/devices to a network. This project focuses on site-to-site; remote access uses clients like WireGuard or OpenVPN.

How do I choose between OpenVPN and WireGuard?

WireGuard: Newer, faster, simpler code, lower latency. OpenVPN: Mature, widely supported, firewall-friendly (uses UDP/TCP 443). For performance: WireGuard. For compatibility: OpenVPN.

Troubleshooting Guide

VPN Tunnel Won't Connect

Symptoms: Tunnel status shows "down" or "inactive"

Solutions:

  • Check IP addresses match on both ends (R1 Tunnel0 vs R2 Tunnel0)
  • Verify routing: show ip route should show tunnel route
  • Check firewall rules allow GRE (protocol 47) or IPSec (UDP 500, 4500)
  • Verify pre-shared keys match (for IPSec)
  • Check interface is up: show interface tunnel0

High Latency Over VPN

Symptoms: RTT > 100ms, slow file transfers

Solutions:

  • Check CPU usage on VPN gateway (encryption is CPU-intensive)
  • Enable hardware acceleration (AES-NI) if available
  • Switch to faster algorithm (ChaCha20 vs AES-256)
  • Check network congestion: show interface
  • Consider WireGuard for better performance

Packet Loss Over Tunnel

Symptoms: Intermittent connectivity, dropped packets

Solutions:

  • Check MTU size (GRE adds 24 bytes overhead): ping -df -s 1472
  • Verify no fragmentation: adjust MTU to 1400 if needed
  • Check for QoS issues on intermediate routers
  • Monitor tunnel statistics: show crypto session brief

Can't Reach Remote Subnet

Symptoms: Tunnel is up but can't ping remote hosts

Solutions:

  • Verify routing on both sides includes remote subnet
  • Check ACL (access control list) allows traffic
  • Verify remote host is on correct subnet (e.g., 192.168.2.0/24)
  • Check firewall on remote host allows ICMP
  • Use traceroute to identify where packets are dropped
