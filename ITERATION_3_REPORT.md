## 2. Practical Work Completed

### 2.1 Ubuntu VM Configuration

An Ubuntu virtual machine was installed and configured in VirtualBox with a bridged network adapter to allow direct communication with the host machine and GNS3 topology. The VM was assigned IP address 192.168.1.33/24 on the enp0s3 interface, confirmed using the `ip addr` command. Internet connectivity was verified by pinging google.com. The AliceInWonderland.txt file was downloaded using wget, renamed, and confirmed present in the home directory using `ls`.

A Python HTTP server was started on the VM using:
```bash
python3 -m http.server 8080
```

This allowed the file to be served over the network to other devices, simulating a real application server within the GNS3 topology.

### 2.2 Real File Transfer Measurement – Baseline Result

To establish a baseline performance metric, the AliceInWonderland.txt file (~170KB) was transferred from the Ubuntu VM (192.168.1.33) to the Windows host machine using PowerShell Measure-Command:

```powershell
powershell -command "Measure-Command { Invoke-WebRequest http://192.168.1.33:8080/AliceInWonderland.txt -OutFile alice_test.txt }"
```

**Baseline Result:**
- Transfer time: 0.287 seconds (287ms)
- File size: ~170KB
- Throughput: ~592 KB/s

This baseline result represents unencrypted direct transfer with no VPN overhead. It serves as the reference point for comparing GRE, GRE+IPSec, WireGuard, and OpenVPN transfer times.

### 2.3 GNS3 Topology – Three Router Design

The topology was extended to include three routers (R1, R2, R3) connected through a central Hub1. R1 and R2 each connect to Ubuntu host machines representing separate local networks. R3 acts as an intermediate transit router, simulating an ISP or external WAN segment between the two sites. A VPCS client (PC1) is also connected to the hub for basic connectivity testing.

This design enables:
- GRE tunnelling between R1 and R2 through R3
- Realistic multi-hop VPN traffic simulation
- Wireshark monitoring of encapsulated packets at the transit point
- File transfer testing between Ubuntu hosts across the tunnel

### 2.4 GRE Tunnel Setup and Verification

A GRE tunnel (Tunnel0) was configured between R1 and R2. Tunnel interfaces were assigned IP addresses and brought up. Verification steps included:
- Confirming tunnel interface status with `show interface tunnel0`
- Verifying routing tables showed remote subnets via the tunnel interface
- Running ping tests between PC1 and PC2 across subnets
- Running traceroute to confirm packets traversed the GRE tunnel path
- Capturing GRE-encapsulated packets in Wireshark showing GRE headers and inner IP payloads

### 2.5 Wireshark Traffic Analysis

Wireshark was used throughout testing to capture and analyse network traffic. Key captures included:
- TCP traffic generated by HTTP file transfers showing source/destination IPs and port numbers
- UDP traffic from DNS queries and ping tests
- GRE-encapsulated packets showing the outer IP header and inner payload
- VPN tunnel traffic filtered by `ip.addr == 172.64.155.209` confirming UDP port 443 encapsulated VPN packets
- ESP-encrypted packets after IPsec activation showing unreadable payloads confirming encryption

Before IPsec activation, ICMP payloads between PC1 (192.168.1.10) and PC2 (192.168.2.10) were visible in plaintext. After IPsec was enabled, the same traffic appeared as encrypted ESP packets — payload unreadable — confirming that IPsec encryption was functioning correctly.

## 3. Traffic Capture and Protocol Analysis

TCP traffic was simulated using HTTP file transfers of AliceInWonderland.txt. UDP traffic was generated using DNS queries and ICMP ping tests. Wireshark confirmed the network handled both TCP and UDP over GRE tunnels, demonstrating multi-protocol flexibility essential for real-world environments supporting web services, VoIP, and real-time data streams.

Encryption was validated by monitoring the AliceInWonderland.txt transfer. Before encryption, file content was visible in packet payloads. Inside the IPsec tunnel, the same payload appeared as random encrypted bytes — unreadable without the decryption key. This confirmed bidirectional encryption was functioning correctly.

## 4. IPsec Implementation

IPsec was deployed over the GRE tunnel in both AH and ESP modes. Wireshark captures confirmed:
- ESP headers present on all tunnel packets
- Sequence numbers and SPI (Security Parameter Index) visible in packet details
- Inner payloads fully encrypted and unreadable
- Correct decryption at the receiving endpoint restoring original data

Encryption algorithm testing compared AES-256 and 3DES, confirming AES-256 provided stronger security with lower computational overhead on modern hardware.

## 6. Performance Testing Results – COMPLETED

### 6.1 Comprehensive VPN Performance Measurements

All VPN configurations were tested with 20 trials each using the AliceInWonderland.txt file (~170KB). Results are presented below:

| VPN Technology | Transfer Time (s) | Throughput (KB/s) | Trials | Overhead vs Baseline |
|---|---|---|---|---|
| **Baseline (No VPN)** | 0.2896 ± 0.0088 | 587 ± 7 | 20 | — |
| **GRE Tunnel** | 0.3084 ± 0.0105 | 551 ± 18 | 20 | +6.5% |
| **GRE + IPSec** | 0.9484 ± 0.0581 | 179 ± 16 | 20 | +227.6% |
| **WireGuard (GNS3)** | 0.5924 ± 0.0320 | 287 ± 12 | 20 | +104.5% |
| **OpenVPN** | 0.7951 ± 0.0322 | 214 ± 8 | 20 | +174.6% |
| **Python VPN (local)** | 0.0094 ± 0.005 | 18,036 | 3 | +1,350% |
| **Pritunl** | 0.305 ± 0.012 | 571 ± 8 | 20 | +6.3% |

### 6.2 Key Findings

**GRE Tunnel Performance:**
- Minimal overhead of only 6.5% compared to baseline
- Transfer time increased from 0.287s to 0.308s
- Throughput reduction minimal (587 → 551 KB/s)
- Demonstrates GRE encapsulation has negligible performance impact
- Suitable for high-performance requirements with basic tunneling

**GRE + IPSec Performance:**
- Significant overhead of 227.6% due to encryption processing
- Transfer time increased to 0.948s (3.3x slower than baseline)
- Throughput reduced to 179 KB/s (69.5% reduction)
- Encryption overhead is substantial but acceptable for security-critical applications
- Consistent results across trials (stdev: 0.0581s) indicate stable encryption performance

**WireGuard Performance (GNS3):**
- Moderate overhead of 104.5% compared to baseline
- Transfer time: 0.592s (2.05x slower than baseline)
- Throughput: 287 KB/s (51.1% reduction)
- Significantly more efficient than OpenVPN and GRE+IPSec
- Modern cryptography implementation provides good security-to-performance ratio

**OpenVPN Performance:**
- High overhead of 174.6% compared to baseline
- Transfer time: 0.795s (2.77x slower than baseline)
- Throughput: 214 KB/s (63.5% reduction)
- More CPU-intensive than WireGuard due to older protocol design
- Still acceptable for general-purpose VPN deployments

**Python VPN (local, Fernet AES-128):**
- Extreme overhead of +1,350% due to Python implementation (interpreted language)
- Baseline: 0.0007s, 261 KB/s → VPN: 0.0094s, 18 KB/s
- Demonstrates real encryption cost in pure Python vs compiled protocols

**Pritunl Performance:**
- Excellent 6.3% overhead, comparable to GRE
- Fast setup (9 min), enterprise features (user mgmt, dashboard)
- Ideal for SMEs

### 6.3 Statistical Analysis

**Consistency Analysis (Standard Deviation):**
- Baseline: 0.0088s (most consistent)
- GRE: 0.0105s (very consistent)
- GRE+IPSec: 0.0581s (consistent, higher variance due to encryption)
- WireGuard: 0.0320s (consistent)
- OpenVPN: 0.0322s (consistent)

All measurements show low variance, indicating reliable and repeatable results across 20 trials.

**Performance Ranking (Best to Worst):**
1. Baseline (No VPN) — 0.2896s
2. GRE Tunnel — 0.3084s (+6.5%)
3. WireGuard — 0.5924s (+104.5%)
4. OpenVPN — 0.7951s (+174.6%)
5. GRE + IPSec — 0.9484s (+227.6%)
