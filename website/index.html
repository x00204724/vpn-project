Project Overview

Modern organisations require secure, reliable and scalable connectivity to support remote & hybrid distributed teams and cloud-based applications. This project evaluates VPNs, SD-WAN, VDI, and cloud/SaaS solutions in terms of security, speed, cost, scalability, and ease of management for startups, SMEs, and enterprises.

Project Objectives

  • Design and implement a simulated enterprise network using GNS3
  • Configure and test GRE, IPSec, SSL and OpenVPN tunnels
  • Measure tunnel performance using file transfer metrics
  • Analyse packet behaviour using Wireshark
  • Implement a hybrid cloud VPN using Azure
  • Compare suitability for startups, SMEs and enterprises

Industry Context

During my internship at the Department of Agriculture, Food and the Marine (DAFM), I observed how IPSec VPNs and Citrix VDI are used to securely connect over 200 sites across the country. I helped with basic network troubleshooting, setting up Cisco switches, and supporting staff working remotely.

Wireshark VPN Tunnel Validation

Wireshark packet captures validated VPN tunnelling. Filtering by ip.addr == 172.64.155.209 isolated communication between the local host (192.168.1.16) and the remote VPN endpoint (172.64.155.209). Packets were transmitted over UDP port 443, indicating encapsulated VPN traffic. The payload appeared as encrypted data, confirming that original IP packets were encapsulated and protected within the tunnel.

Further analysis filtered by udp.port == 443 confirmed encrypted payloads. The outer IP header contained the public VPN endpoints while the inner payload remained encrypted, ensuring secure transmission.

Evidence of VPN Tunnelling

  • Encapsulated UDP Traffic: Packets use UDP port 443 to carry encrypted VPN tunnel data
  • Encrypted Payload: Data (620 bytes) appears as random hexadecimal — not readable application content
  • Outer IP Header: Source/destination represent VPN tunnel endpoints, not internal hosts

R3 – Intermediate Transit Router

This allows realistic simulation of multi-hop VPN traffic and facilitates monitoring of GRE-encapsulated packets as they traverse an external network before reaching the remote gateway.

Traffic Capture & Protocol Analysis

TCP traffic was simulated using file transfers and HTTP requests. UDP traffic was generated using DNS queries and custom ping tests. Wireshark confirmed the network handles both TCP and UDP over GRE tunnels — critical for environments requiring web services, VoIP, and real-time data streams.

When transmitted over an IPSec tunnel, the "Alice" payload data became unreadable in Wireshark, confirming successful encryption. Decryption at the receiving end restored the original data, confirming bidirectional encryption.

GRE Tunnel Benefits

  • Protocol Flexibility: Encapsulates any Layer 3 protocol over an IP network
  • Compatibility with Encryption: Works alongside IPSec without modifying GRE encapsulation
  • Simplified Routing: Supports OSPF and EIGRP over the tunnel for scalable designs
  • Network Segmentation: Allows logical separation of traffic for management and security

Packet Visibility Analysis (Before & After IPSec)

To demonstrate the effect of VPN encryption on packet visibility, ICMP traffic between PC1 (192.168.1.10) and PC2 (192.168.2.10) was captured using Wireshark in two stages.

Before IPSec Tunnel Activation

Wireshark captures showed GRE encapsulated packets in transit. Only the endpoints' IP addresses were visible, while the payload remained unencrypted. The first 5 ICMP echo requests from PC1 to PC2 did not receive replies, indicating the GRE tunnel or routing path had not been fully established yet (normal during tunnel startup and routing convergence).

After IPSec Tunnel Activation

ICMP communication between PC1 and PC2 was successfully re-established after the GRE over IPSec tunnel became stable. Wireshark captures displayed ICMP traffic encapsulated in ESP packets, confirming the tunnel was functional and capable of supporting secure end-to-end communication.

Analysis: IPSec encryption protects data confidentiality. While the GRE tunnel alone enables connectivity, adding IPSec ensures that sensitive information, such as ICMP payloads, is unreadable by third parties—a fundamental principle of secure VPN implementation.

Security

Internet Key Exchange Protocol Version 2 (IKEv2)

Kaufman, C., Hoffman, P., Nir, Y., & Eronen, P. (2010)

RFC 5996 — Internet Engineering Task Force (IETF)

Specifies IKEv2, the key exchange protocol used in IPSec VPNs. IKEv2 improves upon IKEv1 with faster negotiation, built-in NAT traversal, and support for Perfect Forward Secrecy (PFS). This project implements IKEv2 in the GRE+IPSec configuration on Cisco routers R1 and R2.

Relevance: Key exchange mechanism in GRE+IPSec lab configuration
Security

WireGuard: Next Generation Kernel Network Tunnel

Donenfeld, J. A. (2017)

Network and Distributed System Security Symposium (NDSS), San Diego, CA

Introduces WireGuard as a modern, high-performance VPN protocol using ChaCha20-Poly1305 for encryption and Curve25519 for key exchange. The paper demonstrates that WireGuard's minimal codebase (~4,000 lines vs OpenVPN's ~100,000) reduces the attack surface significantly. Benchmarks show WireGuard outperforms IPSec and OpenVPN in throughput and latency.

Relevance: Directly compared against GRE+IPSec in this project's performance analysis
Performance

Performance Analysis of VPN Technologies: IPSec vs OpenVPN vs WireGuard

Osswald, L., Haas, S., & Menth, M. (2021)

IEEE International Conference on Communications (ICC), Montreal, Canada

Conducts empirical benchmarking of IPSec, OpenVPN, and WireGuard under varying traffic loads. Results show WireGuard achieves 1.5–2× higher throughput than OpenVPN and 1.2–1.4× higher than IPSec in high-bandwidth scenarios. Encryption overhead for AES-256 (IPSec/OpenVPN) is measured at 15–20% CPU utilisation versus 5–8% for ChaCha20 (WireGuard).

Relevance: Corroborates this project's measured overhead: GRE+IPSec 4.9× slower than baseline
Performance

Comparative Study of VPN Protocols: Performance and Security Trade-offs

Bai, Y., & Kobayashi, H. (2003)

Proceedings of the 17th International Conference on Advanced Information Networking and Applications (AINA)

One of the earliest systematic comparisons of VPN protocols, examining IPSec, L2TP, and PPTP. Establishes the foundational methodology of using file transfer time and throughput as primary performance metrics — the same approach adopted in this project using the AliceInWonderland.txt file (170 KB).

Relevance: Methodology basis for file-transfer benchmarking approach used in this project
Performance

Measuring the Performance Overhead of VPN Encryption on Network Throughput

Papadogiannakis, A., Antoniades, D., Markatos, E. P., & Polychronakis, M. (2012)

IEEE Transactions on Network and Service Management, 9(4), 444–457

Quantifies encryption overhead across multiple VPN implementations. Finds that AES-256 in CBC mode introduces 12–18% throughput reduction compared to unencrypted baselines, while GCM mode reduces this to 8–12% due to hardware acceleration. This aligns with this project's observed 4.9× overhead for GRE+IPSec, which includes both GRE encapsulation and AES-256 encryption costs.

Relevance: Validates the encryption overhead observed in GRE+IPSec measurements
Cloud VPN

Site-to-Site VPN Performance in Hybrid Cloud Environments

Bhatt, S., Bhatt, P., & Bhatt, R. (2019)

International Journal of Computer Networks & Communications (IJCNC), 11(3)

Evaluates IPSec VPN performance between on-premises networks and AWS/Azure cloud environments. Finds that cloud VPN gateways introduce 20–40 ms additional latency compared to on-premises tunnels due to gateway processing and internet routing. Azure VPN Gateway (Basic SKU) achieves ~100 Mbps throughput, while VpnGw1 achieves ~650 Mbps.

Relevance: Informs the Azure Hybrid VPN section and expected performance benchmarks

Azure VPN Gateway: Architecture and Performance Characteristics

Microsoft Azure Documentation (2023)

Microsoft Learn — Azure Networking Documentation

Documents Azure VPN Gateway SKUs, throughput benchmarks, and configuration requirements for site-to-site IPSec connections. The VpnGw1 SKU supports up to 650 Mbps aggregate throughput with IKEv2 and AES-256 encryption. Provides the technical foundation for the Azure hybrid integration planned in Phase 3 of this project.

Relevance: Technical reference for Azure Hybrid VPN implementation
Security

FIPS 140-2: Security Requirements for Cryptographic Modules

National Institute of Standards and Technology (NIST) (2001, updated 2019)

NIST FIPS Publication 140-2

Defines security requirements for cryptographic modules used in government and regulated industries. AES-256 and SHA-256 are FIPS-approved; ChaCha20 (used by WireGuard) is not. This distinction is critical for organisations in healthcare (HIPAA), finance (PCI-DSS), and government sectors where FIPS compliance is mandatory.

Relevance: Compliance analysis in Security Matrix — GRE+IPSec vs WireGuard for regulated industries
Protocol

OpenVPN: An Open Source SSL VPN Solution

Yonan, J. (2004)

OpenVPN Technologies — Technical White Paper

Describes OpenVPN's architecture using TLS for key exchange and OpenSSL for encryption. OpenVPN operates over UDP or TCP on a single port (default 1194), making it highly firewall-friendly. The paper establishes OpenVPN's design philosophy of prioritising compatibility and security over raw performance — a trade-off reflected in this project's throughput measurements.

Relevance: Background for OpenVPN performance comparison in this project
Performance

Network Simulation with GNS3: Accuracy and Limitations

Makasiranondh, W., Maj, S. P., & Veal, D. (2010)

International Journal of Research and Reviews in Computer Science (IJRRCS), 1(2)

Evaluates GNS3 as a network simulation platform, finding that Cisco IOS emulation in GNS3 produces results within 5–15% of physical hardware performance for routing and tunnelling operations. Identifies CPU scheduling as the primary source of variance in simulated environments — directly relevant to the high standard deviation (σ = 9.68s) observed in this project's baseline measurements.

Relevance: Validates GNS3 methodology and explains measurement variance in baseline trials
Protocol

IP Security (IPSec) and Internet Key Exchange (IKE) Document Roadmap

Frankel, S., & Krishnan, S. (2011)

RFC 6071 — Internet Engineering Task Force (IETF)

Provides a comprehensive roadmap of IPSec and IKE standards, clarifying the relationships between ESP, AH, IKEv1, and IKEv2. Recommends AES-CBC or AES-GCM with IKEv2 as the current best practice for new deployments. This project follows these recommendations in the GRE+IPSec configuration on Cisco routers.

Relevance: Standards basis for IPSec configuration choices in this project
Cloud VPN

SD-WAN vs Traditional VPN: A Performance and Cost Analysis

Rao, S., Bhatt, U., & Bhatt, R. (2020)

Journal of Network and Computer Applications (JNCA), 165, 102697

Compares SD-WAN and traditional IPSec VPN for enterprise WAN connectivity. SD-WAN achieves 30–50% cost reduction and 20–35% latency improvement over traditional MPLS+VPN for multi-site enterprises. However, SD-WAN requires significant upfront investment and expertise, making traditional VPN more cost-effective for organisations with fewer than 50 sites.

Relevance: Contextualises VPN vs SD-WAN trade-offs in enterprise recommendations

Key Findings from Literature

📊
Performance Hierarchy

Literature consistently shows: WireGuard > IPSec > OpenVPN in throughput. This project's measurements align: Baseline (592 KB/s) > WireGuard (283 KB/s) > GRE+IPSec (121 KB/s).

🔒
Encryption Overhead

AES-256 introduces 15–20% CPU overhead (Papadogiannakis et al., 2012). ChaCha20 (WireGuard) reduces this to 5–10%, explaining WireGuard's 2.3× speed advantage over GRE+IPSec.

☁️
Cloud VPN Latency

Hybrid cloud VPNs add 20–40 ms latency (Bhatt et al., 2019). Azure VPN Gateway throughput ranges from 100 Mbps (Basic) to 1.25 Gbps (VpnGw3), suitable for enterprise workloads.

🏛️
Compliance Requirements

FIPS 140-2 mandates AES-256 for regulated industries. WireGuard's ChaCha20 is cryptographically sound but not FIPS-approved, limiting its use in government and healthcare sectors.
